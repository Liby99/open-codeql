# L2 — Core QL (MIR)

- **Layer:** L2
- **Implementation crates:** `ocql-mir` (active), `ocql-lir` (currently a stub)
- **Status:** Provisional. The MIR shape and lowering rules are settled in code; this document codifies them.
- **Version:** 0.1

## 1. Purpose

L2 is **Core QL**: a small, predicate-centric Datalog dialect that is the
target of L1 lowering and the input to L3 evaluation. It strips away the
surface conveniences of QL (classes, modules, aggregations as first-class
values, recursive class membership) into a flat program of named
predicates and atoms.

L2 is the **principal stable contract** of open-codeql. New surface
syntax in L0/L1 must lower into existing L2 constructs; new evaluation
features in L3 must compile from existing L2 constructs.

## 2. MIR program

```
MirProgram = {
   predicates : [MirPredicate],
}

MirPredicate = {
   name      : QualifiedName,             // dotted, post-qualification
   params    : [(VarName, MirType)],
   result    : Option<(VarName, MirType)>, // for predicates that return
   body      : MirFormula,
   stratum   : Option<u32>,                // assigned at L2→L3
   is_dispatch : bool,                     // generated by class lowering
}

MirFormula = And [MirAtom]
           | Or  [MirFormula]

MirAtom = Scan         { pred, args : [MirTerm] }
        | NegScan      { pred, args : [MirTerm] }
        | Guard        { left, op, right }
        | Assign       { var, expr : MirArith }
        | Aggregate    { result_var, fn, sub_pred, group_by, agg_var }
        | TypeCheck    { var, ty }

MirTerm  = Var(name) | Const(Value) | Wildcard
```

`MirType` ranges over the L1 type space, but L2 evaluation treats types as
**opaque tags**: a `TypeCheck` atom is a call to the corresponding
characteristic predicate.

## 3. Lowering from L1

### 3.1 Classes → characteristic predicates

A class `C` lowers to one **characteristic predicate** named after the
class. Its body is the conjunction described in L1 §3.2 (parents,
`instanceof`, explicit characteristic predicate). Member predicates of `C`
become free predicates with an explicit `this` parameter and a name of the
form `C#member`.

### 3.2 Modules → name qualification

After L1 instantiation, every predicate is reachable by a fully qualified
dotted name. L2 has no module construct: the only naming structure is the
`QualifiedName` of `MirPredicate`.

### 3.3 Newtypes — Provisional

Newtype lowering is currently **not implemented** (`ocql-mir/src/lower.rs`
near line 232). The intended rule is:

- A newtype `T = Var1(...) | Var2(...) | ...` becomes one
  characteristic predicate per branch (`T#Var1`, …) plus a union
  predicate `T(x) ↔ T#Var1(x) ∨ T#Var2(x) ∨ …`.

### 3.4 Quantifiers and negation

| Surface | Lowering |
|---------|----------|
| `exists(t v \| P(v))` | New aux predicate over the free vars of `P`; head is `Scan(aux, free-vars)`. |
| `forall(t v \| G(v) \| B(v))` | `not exists(t v \| G(v) and not B(v))`. |
| `forex(t v \| G(v) \| B(v))` | `exists(...)` ∧ `forall(...)`. |
| `not P(...)` (P atomic) | `NegScan(P, args)`. |
| `not F` (F compound) | New aux predicate for `F`; outer `NegScan(aux, …)`. |

### 3.5 Aggregations

```
result = agg(t v | body | agg-expr)
   ↓
   aux(group-by-vars, agg-var) :- body, agg-var = agg-expr
   Aggregate { result, fn=agg, sub_pred=aux, group_by=…, agg_var=… }
```

The `Aggregate` atom names the auxiliary predicate so that L3 can evaluate
it under standard semi-naive rules and apply the aggregation function as
a post-step.

> **Conformance gap:** `concat` separators and `order by` clauses on
> `concat`/`rank` are parsed and lowered syntactically but not faithfully
> evaluated. See L3 §6.2.

### 3.6 Closure operators (`+`, `*`)

```
e+(args)            // transitive closure of e
e*(args)            // reflexive transitive closure of e
   ↓
   aux(x, y) :- e(x, y)
   aux(x, y) :- aux(x, z), e(z, y)
   // for `*`, add aux(x, x).
```

> **Conformance gap:** the current lowering is partial; some uses still
> bottom out as a TODO (`ocql-mir/src/lower.rs` near line 879).

### 3.7 Select queries

A top-level `from … where … select …` lowers to a single predicate
`select_result/n` whose head holds the projected columns. If `order by`
or column aliases are present, a second predicate `select_output/n`
projects the user-visible columns.

## 4. Stratification

L2 admits a program iff its **predicate dependency graph** is stratifiable
under negation:

- Edge from `p` to `q` for each occurrence of `q` (positive or negative)
  in `p`'s body.
- The edge is `Negative` if it crosses a `NegScan` or appears under
  `forall` / aggregation.
- Strongly connected components are computed by Tarjan's algorithm.
- A program is stratifiable iff no SCC contains a Negative edge.

Stratification assigns each predicate a stratum number; L3 evaluates
strata in topological order.

`ocql-engine/src/stratify.rs` is the binding implementation.

## 5. LIR — Provisional

The L2 design originally envisaged a separate **LIR** layer of
relational-algebra operators (Join, Project, Select, Union, Difference,
Fixpoint, Aggregate, FlowJoin) sitting between MIR and the engine. In the
current code, MIR rules are evaluated directly by `ocql-engine`; the LIR
crate is a stub.

This spec retains LIR as a **Provisional** sub-layer with the following
contract:

- LIR is a normal form for L3 evaluation. Each MIR predicate compiles to a
  single LIR expression tree.
- LIR is the natural injection point for query optimization (join
  ordering, magic-set transforms, predicate pushdown). It is also the
  natural place to materialize **flow joins** for L3 §5.

Until LIR is built, the L2 → L3 boundary is `MirProgram` itself, and the
engine is permitted to do its own evaluation planning.

## 6. Determinism

L2 is deterministic up to:

- The order in which MIR predicates appear in the program (irrelevant —
  evaluation is set-based).
- The order of disjuncts in an `Or` formula (irrelevant — evaluation
  produces sets).
- The order of conjuncts in an `And` formula (irrelevant for results,
  relevant for performance — see LIR planning above).

Two L1 programs that lower to MIR programs differing only in those orders
must produce the same L3 results.

## 7. Conformance tests

For each lowering rule in §3:

- One **shape test** verifying the MIR S-expression matches a snapshot.
- One **behavioural test** verifying that the L3 evaluation of the lowered
  program yields the same tuples as a hand-written equivalent MIR program.

The shape and behavioural tests are independent: passing one without the
other indicates either a stale snapshot (shape only) or a hidden coupling
in the engine (behaviour only).

## 8. Open questions

- **Reify LIR:** is LIR still earning its place, or should we collapse to
  an MIR-only IR and put the planner inside `ocql-engine`?
- **Aggregate `order by` and `concat` separators (§3.5):** should L2 carry
  them as `Aggregate` parameters (current direction) or lower them as
  per-aggregation auxiliary structure?
- **Newtype lowering (§3.3):** does the union predicate need a tag column
  to disambiguate variants at use sites?
